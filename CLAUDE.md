# WAZA KIMURA — 開発ルール（Claude用）

BJJ動画管理アプリ。**ユーザーの実データ（動画・カスタムビュー・ノート・タグ）を保持する本番運用中のツール**。
データはFirestore（クラウド・複数デバイス同期）と localStorage（端末ローカルキャッシュ）の二層で保持される。

## 🛑 最優先ルール: データ層の変更は高リスクとして扱う

過去に「ログアウト時にカスタムビューを非表示」という些細な見た目変更が、
`_views`空化→`saveUserSettings`→Firestore上書き→**全デバイスでカスタムビュー消失**を引き起こした（v52.541）。
教訓: *仕組みが動いた ≠ 結果が正しい*。局所のテストではなく**システム全体の帰結**を見る。

### 高リスク対象（これらに触れる変更は下記プロトコル必須）
- `js/firebase.js`（特に `saveUserData` / `saveUserSettings` / `loadUserSettings` / `onAuthStateChanged`）
- localStorage キーへの書き込み・**削除**（`wk_cv_views`, `wk_tagGroups`, `wk_*`, `waza_*` 等）
- `_views` / `window._cvViews` / `tagGroups` など**クラウド同期される状態**の書き換え
- あらゆる `removeItem` / `delete` / 配列の空化 / 既存データの上書き
- 認証状態（ログイン/ログアウト）に連動する処理

### コミット前プロトコル（必須）
1. **全データ経路を書き出す**: その変更がどの状態を変え、それが localStorage / Firestore / 他デバイス にどう波及するか。
2. **最悪ケースを問う**: 「これは他のデバイス/セッションに存在するデータを、消す・空で上書きする可能性があるか？」
   - YES の可能性が少しでもあれば、**空や欠損で上書きしない安全方向の実装**にする（非空→空の上書きは原則禁止）。
3. **メリットとブラスト半径を天秤にかける**。見た目改善程度のためにデータ破壊リスクを取らない。
4. 検証は「仕組みが動く」ではなく「**データが安全な結果になる**」を確認する。

## 🛑 ルール2: マージ運用 — オーナー承認済みで毎回 main へマージしてよい

オーナー（okujournal@gmail.com）から「今後はデータ層を含め確認不要で毎回本番マージしてよい」と恒久承認を得ている（2026-06-28）。
したがって通常の機能/UI変更も**高リスク対象に触れる変更も、ユーザー確認を取らずに main へマージ・デプロイしてよい**。

ただし承認されたのは「都度の確認を省くこと」であって「安全確認の省略」ではない。以下は引き続き必須:
1. **ルール1のコミット前プロトコル（全データ経路の書き出し・最悪ケース検証・非空→空の上書き禁止）は毎回必ず実施する**。安全方向の実装を徹底する。
2. データ層に触れる変更をマージしたら、**何を変えたか・どのデータに波及するかを返信で明示**する（事後報告）。
3. 破壊的変更が避けられない場合のみ、マージ前に一声かける（恒久承認の例外）。

## 🌐 多言語（i18n）ルール — 英語ユーザーも本番利用中
仕組み: `js/i18n.js` が表示層を自動翻訳（EN時のみ・MutationObserver）。**データは常に日本語のまま保存**し、表示だけ訳す。
ユーザーデータ（動画タイトル・メモ・ノート名・プレイリスト名等）は**絶対に翻訳しない**（contenteditable/入力値はエンジンが自動でスキップ）。

### 新しいUI文言を追加するとき（必須・同じコミットで）
1. **固定文** → `STATIC_AUTO` に `'日本語':'English'` を追加。
2. **数値入りテンプレ文**（`${n}本を追加` 等）→ `TEMPLATE_AUTO` に追加。キーは**数字列をすべて `#` に置換**したもの（例: `'✅ #件を追加しました':'✅ Added #'`。`MP4`の4や`1つ`の1も `#` になる点に注意）。
3. **文字列補間文**（名前が入る等）→ `AUTO_PATTERNS` に正規表現で追加。関数は `null` を返すと「このパターンでは訳さない」。
4. タグ分類（POSITIONS/カテゴリ/TB/TECHNIQUE_BUILTIN）は実行時に自動登録されるので何もしなくてよい。

### 確認方法
- `node tools/i18n-check.mjs` … コード全体から日本語を抽出しエンジンに通して未訳を列挙（ブラウザ不要）。
  既知の許容残: console.log・AIルール文・管理画面・連結断片（実行時はパターンが拾う）。
- 実行時の取りこぼしは英語モードで一通り操作後、コンソールで `wkI18nMissing()`。

## 不具合対応の原則
- **まず“見える化”してから直す**。原因を推測で順番に潰さない。最初の一手は観測（実エラー/状態を画面やログに出す）。
- 各メッセージを独立タスクとして処理せず、アプリ全体・データモデル・リスク面を握り続ける。

## デプロイ / バージョン
- 本番は **Cloudflare Worker**（`waza-kimura.okujournal.workers.dev`、`_worker.js` + 静的アセット）。`/api/*` は `_worker.js` が処理。
- 変更ごとに `index.html` の `<title>WAZA KIMURA vXX.XXX</title>` を上げる（デプロイ確認用）。
- 作業ブランチ: `claude/epic-ride-er1IM`。main push で本番デプロイ。
- PWAキャッシュがあるため、ユーザーには反映確認時にハードリロードを案内する。
